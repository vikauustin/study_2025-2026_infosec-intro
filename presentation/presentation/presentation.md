## Основные этапы(1-3)

:::::::::::::: {.columns align=center}
::: {.column width="70%"}

Этап 1. Базовые понятия: субъект, объект, права, матрица доступа

Этап 2. DAC (избирательный):

владелец решает, кто получает доступ

примеры: chmod (Linux), разрешения NTFS (Windows)

гибко, но небезопасно

Этап 3. MAC (мандатный):

система задаёт метки (секретно, совершенно секретно)

примеры: SELinux, FreeBSD MAC, военные системы

безопасно, но негибко

## Основные этапы(4-6)

:::::::::::::: {.columns align=center}
::: {.column width="70%"}

Этап 4. RBAC (ролевой):

права через роли (бухгалтер, менеджер, админ)

примеры: Active Directory, 1С, базы данных

Этап 5. ABAC (атрибутный):

доступ по атрибутам (должность, время, место, устройство)

примеры: AWS IAM, Azure AD

Этап 6. POLP (наименьшие привилегии):

давать ровно столько прав, сколько нужно

примеры: UAC (Windows), sudo (Linux)

## Анализ и практическая значимость достигнутых результатов
**Результаты сравнения:** DAC: гибкий, но низкая безопасность, MAC: безопасный, но негибкий, RBAC: золотая середина, ABAC: гибкий + безопасный, но сложный, POLP: обязательное дополнение к любому методу

**Практические рекомендации:**Дом / малый офис → DAC + POLP, Организация с иерархией → RBAC + POLP, Секретные данные → MAC + POLP, Облака / распределённые системы → ABAC + POLP

## Общее заключение и выводы

**Выводы:**

1. Универсального метода не существует

2. На практике часто используются гибридные модели (Windows: DAC + RBAC, SELinux: MAC поверх DAC)

3. ABAC + POLP — современный стандарт для облачных и распределённых систем

4. Правильный выбор метода — основа безопасности любой информационной системы

*Итог:* метод выбирается под конкретную задачу с учётом требований к безопасности, сложности и среды
